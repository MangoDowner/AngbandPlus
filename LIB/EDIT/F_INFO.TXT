# File: f_info.txt

# This file is used to initialize the "lib/raw/f_info.raw" file, which is
# used to initialize the "terrain feature" information for the Angband game.

# Do not modify this file unless you know exactly what you are doing,
# unless you wish to risk possible system crashes and broken savefiles.
# After modifying this file, delete the "lib/raw/f_info.raw" file.

# Note that the terrain feature are grouped into very regular groups,
# such that each of the bits in the feature type conveys information.

# Note that this is no longer the case, but we go to extraordinary lengths to
# retain the significance of the LOS bit (bit 5).

# Note that terrain feature zero contains the "darkness" picture.

# Feature flags
# -------------

# Features now have flags, which are broken up into 2 sets of 32 flags, plus 2
# dummy flags (LOS and POWER_X). Feature flags simplify handling of transition
# of one feature to another (OPEN, CLOSE, HURT_FIRE etc.) and allow the cmd2.c
# code to be written cleaner and more flexibly.

# Note that LOS is implied implicitly by the 5th bit of the feature number. We do
# this for backwards compatibility with current save files.

# It is going to be replaced by using the CAVE_WALL bit set in the cave_info.
# This will allow us to have 512 features, which we would really like. We could
# possibly expand this to 1024 features with replacing CAVE_ICKY with a feature
# flag.

# The flags (2 x 32 bits).

# Vanilla flags.

# LOS:          LOS allowed through this location. DUMMY FLAG.
# POWER_X:	Power of bash/open/tunnel resistance. DUMMY FLAG.

# PROJECT:	Project allowed through this location.
# MOVE:		Normal movement allowed through this location.
# PLACE:	Place monsters at this location.
# DROP:		Drop items in this location.
# SECRET:	Replace with another feature when found/walked over.
# NOTICE:	Notice this feature while running.
# REMEMBER:	Remember this feature.
# OPEN:		Can open this feature.
# CLOSE:	Can close this feature.
# BASH:		Can bash this feature.
# SPIKE:	Can spike this feature.
# DISARM:	Can disarm this feature.
# ENTER:	Can enter this feature (shop)
# TUNNEL:	Can tunnel this feature.
# STREAMER:	Place as streamer through dungeon.
# HAS_ITEM:	Feature has item.
# HAS_GOLD:	Feature has gold.
# DOOR:		Detected by detect doors/destroyed by destroy doors. Tunnel power of 20.
# TRAP:		Detected by detect traps/destroyed by destroy traps.
# STAIRS:       Detected by detect doors/destroyed by destroy stairs.
# GLYPH:	Destroyed by powerful creatures.
# LESS:         Not allowed in town.
# MORE:		Not allowed on quest levels.
# RUN:		Can run over.
# FLOOR:	Can create floors, traps, doors here.
# WALL:		Blocks flow and treated as WALL by magic mapping/enlightenment.
# PERMANENT:	Never destroy/modify this location.
# INNER:	Inner wall of room - don't overwrite with corridor.
# OUTER:	Outer wall of room - OK to overwrite with corridor/turn to door.
# SOLID:	Outer wall of room - not OK to overwrite with corridor/turn to door.
# HIT_TRAP:	Apply blow/spell only when first moved onto.
# ATTR_LITE:	Attribute is modified by torchlight/blindness/not seen.

# Non-vanilla flags

# BRIDGE:	Bridge this feature when generating a corridor here.
# RIVER:	Place as river through dungeon.
# LAKE:		Place as lake in dungeon.
# GLOW:		Lights itself and adjacent locations.
# BRIDGED:	Take blows from mimiced feature, unless choose to go swimming/burrowing.
# COVERED:	Take blows from mimiced feature, unless choose to go swimming/burrowing, then no LOS.
# WATER:	Never generate on ice, acid, oil or lava level. Mark as water level.
# LAVA:		Never generate on water, acid, oil or ice level. Mark as lava level.
# SHALLOW:	Double weight of load. Chance of sinking in feature and losing move.
# DEEP:		Triple weight of load. Chance of sinking in feature and losing move.
# FILLED:	Quadruple weight of load. Chance of sinking in feature and losing move.
# HURT_ROCK:    Destroyed by transmute rock to mud/rock remover.
# HURT_FIRE:	Destroyed by fire balls/bolts/breath.
# HURT_COLD:	Destroyed by frost balls/bolts/breath.
# HURT_ACID:	Destroyed by acid balls/bolts/breath.
# ICE:		Never generate on water, acid, oil or lava level. Mark as acid level.
# ACID:		Never generate on ice, water, oil or lava level. Mark as acid level.
# OIL:		Never generate on ice, acid, water or lava level. Mark as oil level.
# CHASM:	Replace with chasm if surrounded by CHASMs or chasm edge if adjacent to non-CHASM, non-BRIDGED. 
# CAN_CLIMB:    Adjacent CAN_CLIMB creatures treated as CAN_FLY.
# CAN_FLY:	Creatures with CAN_FLY/MUST_FLY may enter and treated as flying.
# CAN_SWIM:	Creatures with CAN_SWIM/MUST_SWIM may enter, treated as swimming and ignore WATER blows.
# CAN_PASS:     Creatures with PASS_WALL may enter.
# CAN_TINY:	Creatures with TINY may enter.
# CAN_DIG:	Creatures with CAN_DIG may enter, treated as burrowing and ignore WATER blows.
# HIDE_ITEM:	Objects in this location only memorised if detected/walked on.
# HIDE_DEEP:	Creatures in this location only seen/projectable if flying.
# HIDE_SWIM:	Creatures in this location only seen/projectable if not swimming/adjacent.
# HIDE_DIG:	Creatures in this location only seen/projectable if not burrowing/adjacent.
# KILL_HUGE:	Destroyed by powerful attacks.
# KILL_MOVE:	Destroy feature if any creature walks/burrows onto it.

# New feature fields
# ------------------

# K:DEFAULT:NEW_FEATURE
# K:ACTION:NEW_FEATURE
# Change the feature to NEW_FEATURE if ACTION applied to it, and use DEFAULT if action not listed.

# This works with various flags to handle state changes to an object. Note that we can apply state
# changes under some circumstances (such as BASHing features, which 50% of the time are OPENed instead)
# even when a feature does not support that flag. So it is important to have a DEFAULT type in many
# instances.

# We only support up to 4 state changes, so DEFAULT is needed to handle many features because of this.

# W:LEVEL:RARITY:PRIORITY:EDGE
# Level indicates level that item should be first encountered (Unless out of depth), rarity
# is a comparitive rarity of the item, priority is priority for generating map views, and
# edge indicates what feature is generated adjacent to the feature when it is placed.

# We use the same distribution code as objects so that features can be expected (Roughly) to be encountered
# with similar frequency as objects of the same rarity. However, this gets skewed heavily, by various flags.

# T:EFFECT:DICEdSIDES
# Trap effects applied continuously or as a trap. Traps are applied as blows, with the implicit blow method
# oF RBM_TRAP and blows now use code in common with projection code. This allows for many more blow and trap
# types than previously.

# S:SPELL
# Spell effect applied continuously or as a trap (With HIT_TRAP)

# Additional features
# -------------------

# Additional features include the following:

# 1 replacement feature to handle case of unknown jammed door.

# 2 new features to correctly handle case of rubble with hidden items, and rubble with known items.

# 39 new traps, including 4 new types (magic symbol, stone visage, eldritch sign and trip wire) and expanded
# previous types (deep pits, summoning runes of various flavours).

# 13 trapped doors, including unknown trapped doors. We replace a trapped door with the trap type whenever it
# is triggered to save having a door trap for each floor trap. They are treated as locked doors otherwise.

# 1 new streamer (SANDSTONE) which never has hidden or known treasure.

# The remaining new features are treated as terrain features.

# Terrain features
# ----------------

# Terrain features are generated as lakes or rivers, or as a possible replacement for a lake or
# river or when two different terrain types overwrite each other. Zero, one or more terrain features
# are generated per level. Terrain features are generated before placement of rooms or corridors.

# A terrain feature with a LAKE flag generates a lake when selected. A terrain feature with a RIVER flag
# generates a RIVER when selected. A terrain feature with both flags generates both types when selected.

# A river grid is surrounded by grids containing the feature's edge terrain. A river grid is always placed
# adjacent to a previous river grid. A river tends to travel in one direction in the dungeon and stops when
# it reaches the edge of the dungeon.

# A lake grid is surrounded by lake grids, which are edged by grids containing the feature's edge terrain.
# Lake grids are scattered through the dungeon, adjacent to each other, until the maximum number of lake grids
# are placed on non-lake grids. Lake grids may be randomly replaced by the edge terrain, which means
# that a lake grid is not guaranteed to be reachable. XXX XXX XXX

# Rivers and regular lakes are guaranteed to be intersected by at least one corridor, and prevent rooms being
# placed in the same 11x11 location. Big lakes, which occur more frequently lower in the dungeon, may be have
# rooms placed over them, and may have parts of the lake cut off by room placement so that they become
# unreachable. XXX XXX XXX

# Corridors bridge terrain that has BRIDGE flags and tunnel through terrain that has TUNNEL flags. This is
# done to guarantee that there is always a safe path from room to room, even if that safe path requires
# walking through shallow acid. Terrain that is neither bridged or tunnelled through must otherwise be
# crossable.

# Deep terrain comes in two varieties, hidden deep terrain and known deep terrain. Known deep terrain is crossed
# by stone or ice bridges. Hidden deep terrain is crossed by paths of shallow terrain. Hidden deep terrain
# should always be detectable as a secret, so that the paths of shallow terrain may be located and navigated
# across. Great care should be made to never mix known deep and either shallow or hidden deep terrain so that
# we don't get 'breaks' in bridges.

# The base of falls are an exception. They are known deep terrain, that we never bridge. They should be
# uncommon, even when generated, because the majority of their terrain will be hidden deep terrain. They are
# surrounded by shallow terrain, so can always be navigated around.

# Some terrain types overwrite other terrain types with intermediate terrain types. This is determined in
# generate.c but should be replaced with a smart algorithm because there are O(n^2) combinations of terrain.

# Some terrain types replace themselves with different terrain when placed. This is determined in generate.c.

# Trails may be made through terrain of various sort by the player and monsters crossing that terrain. This
# may provide indirect evidence of invisible monsters, and movement of out of line of sight monsters.

# Covered and deep terrain of various types can hide monsters. They can pop-up or bash up from this terrain
# to cast spells and attack the player, and remain visible until they move again.

# We do not have any terrain that only causes nuisance damage, because this would allow the player to
# take unnecessary advantage of such terrain, either by harassing monsters from it, or weakening or killing
# monsters through exploiting the movement algorithms.

# We don't make ice floors and geothermal floors damage the player. However, we do need to distinguish when
# there exists a terrain that a monster may be hidden under, and retain these terrain types for this
# possibility.

# We need to have a 1:1 cave type or we generate lava at low levels. We use granite for this.

# We generate magma, quartz and sandstone cave types without treasure to add some variety to dungeon streamers.

# Limestone is actually a cave type with floor space in it, and rarely occuring limestone formations, which
# can be tunnelled easily.

# Oil and coal catch fire when hit with fire attacks. We need some way of spreading this fire, creating smoke
# and so forth. Currently we don't support deep oil. XXX XXX XXX

# Ice is transparent walls of various types. However, we allow the player to block line of sight
# through ice to save them from distant monsters casting LOS spells at them. This is possible by huge ball
# spells of various sorts, which cause enough damage to crack the ice.

# Chasms use the CHASM flag to considerably neaten themselves up.

# Note that even shallow lava is dangerous (and bridged) and there is no shallow BOILING MUD/WATER.

# Note that we use WATER | LAVA | ICE | ACID | OIL to mark a level as such, and prevent
# features marked with other types, being generated on the same level. Note that various WATER types are
# actually marked as LAVA or ICE types instead, to force generation of boiling or icy water only on the
# particular levels.

# Note that we use WATER | LAVA | ICE | ACID | CHASM to favour generation of creatures suitable
# for that level, ahead of those that are not. This is a hack. XXX XXX

# Note that we have to prevent monsters flowing whenever there is terrain that they cannot flow over within
# the flow space. We use flow_can_fly etc. flags to do this with. We also allow more monsters to move on
# terrain than are technically immune to the damage. We allow IM_FIRE & CAN_SWIM monsters on BWATER blows,
# IM_FIRE and CAN_DIG monsters on BMUD blows. There is no ICE damage that can be caused by floor terrain
# currently. We allow IM_FIRE and BR_FIRE monsters on LAVA. Otherwise we would have to check monsters for
# elemental type and prefix, which adds expense to the flow computation.

# In order to allow this, we must make sure we never damage a monster that can flow through a particular
# terrain type. We make HIT_TRAPs an exception, because most monsters can DISARM or destroy the traps. 

# If a terrain has BLOWS and is not HIT_TRAP then the terrain does damage when moved onto
# and every (dungeon) turn (Actually 10 turns).

# Note that the graphics have expanded but consistant meaning, so that
# "%" is a mineral vein/cracks, ":" is a known deep feature (with rubble 
# being 'deep' open floor), ">" is a chasm/fall/stairs and "<" is stairs/base of falls.
# "#" is consistently a wall feature, passable by CAN_SWIM/CAN_DIG/CAN_PASS only (unlike other
# varients). ";" is either glyph or vent, ''' is an open door or bridge.

# Note that the colours of various features are regularised, although in many instances
# different from creatures. It should provide pretty dungeon pictures though.

# 'w' is used for granite/quartz, 'W' is used for limestone/steam, 'y' for sand/sand stone,
# 'D' for oil/coal, 'b' for water, 'B' for ice,  'g' for acid, 'G' for gas,
# 'R' for lava, 'o' for boiling mud/burning oil, 'v' for boiling water/geothermal
# 's' for broken/missing floor/magma.

# Note that 'r' is not used as it is very hard to distinguish from 'R' (lava) particularly for
# '.' and ':' graphics. Note however that lava is the only feature that GLOWs so should be
# much more easily recognisable.

# Note we also use 'o' to mark trails of various sorts through sand/mud/earth. This doesn't
# confuse the borg because of a couple of minor graphical switches (burning oil always 'looks'
# deep).

# All the traps/trapped doors of a particular appearance have the same colour. We don't
# want to have traps in 'U' because trapped doors will be indistinguishable from the plain
# type.

# We don't mimic traps so that particular users can mark traps with a particular style,
# to indicate 'deadly' traps if required.
